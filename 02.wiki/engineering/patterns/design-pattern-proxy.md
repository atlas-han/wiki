# 프록시 (Proxy)

다른 객체에 대한 대체물 또는 자리표시자를 제공해, 원래 객체에 대한 접근을 제어하고 요청 전후에 부가 동작을 끼워 넣는다. [[design-patterns]]의 구조 패턴에 속한다.

## 문제

데이터베이스 연결처럼 방대한 시스템 자원을 소비하는 무거운 객체가 있는데, 이 객체가 항상 필요한 것은 아니라고 하자. 지연 초기화(필요할 때만 생성)로 해결할 수 있지만, 그 코드를 모든 클라이언트에 넣으면 중복이 발생한다. 게다가 타사 라이브러리의 폐쇄된 클래스라면 그 안에 코드를 직접 추가할 수도 없다.

## 해결책

원래 서비스 객체와 동일한 인터페이스를 구현하는 프록시 클래스를 만든다. 클라이언트는 서비스 대신 프록시를 사용하고, 프록시는 요청을 받으면 실제 서비스 객체를 생성·관리하며 작업을 위임한다. 이 과정에서 프록시는 위임 전후에 지연 초기화, 접근 제어, 캐싱, 로깅 같은 부가 작업을 수행한다. 클라이언트는 자신이 프록시를 쓰고 있다는 사실조차 모를 수 있다.

## 실세계 비유

신용카드는 은행 계좌의 프록시이고, 은행 계좌는 현금의 프록시다. 신용카드와 현금 모두 결제라는 동일한 인터페이스를 구현하며 결제에 쓸 수 있다. 고객 입장에서는 현금을 직접 들고 다니지 않고도 카드라는 대리자를 통해 같은 일을 더 편리하고 안전하게 처리한다.

## 적용 가능성

- 지연 초기화(가상 프록시): 무거운 서비스 객체를 늘 띄워 두는 대신 실제로 필요할 때만 생성
- 접근 제어(보호 프록시): 특정 자격을 가진 클라이언트만 서비스 객체를 사용하도록 제한
- 원격 서비스의 로컬 실행(원격 프록시): 네트워크 통신 세부사항을 대신 처리
- 요청 로깅: 서비스로 가는 요청 기록을 남길 때
- 요청 결과 캐싱: 반복되는 요청의 결과를 저장해 재사용
- 스마트 참조: 사용되지 않는 거대한 객체의 수명 주기를 관리

## 장단점

**장점**
- 클라이언트가 모르는 사이에 서비스 객체를 제어할 수 있다
- 클라이언트가 신경 쓰지 않아도 서비스 객체의 수명 주기를 관리할 수 있다
- 서비스 객체가 아직 준비되지 않았거나 없어도 프록시가 동작할 수 있다
- 개방/폐쇄 원칙: 서비스나 클라이언트를 바꾸지 않고 새 프록시를 도입할 수 있다

**단점**
- 새 클래스가 많이 도입되어 코드가 복잡해질 수 있다
- 서비스의 응답이 한 단계 거쳐 지연될 수 있다

## 다른 패턴과의 관계

- [[design-pattern-adapter]]는 다른 인터페이스를 제공하지만, 프록시는 원본과 동일한 인터페이스를 제공한다.
- [[design-pattern-decorator]]와 구조는 같지만 목적이 다르다. 데코레이터는 기능을 향상(확장)하고, 프록시는 접근을 제어한다. 또 프록시는 보통 서비스 객체의 수명 주기를 스스로 관리하지만, 데코레이터는 클라이언트가 그 구성을 제어한다.
- [[design-pattern-facade]]와 유사하게 복잡한 대상을 가리지만, 퍼사드는 자체 인터페이스를 제공하며 서비스 객체와 상호 교환되지 않는 반면 프록시는 서비스와 같은 인터페이스라 교체 가능하다.
