### LLM 에이전트가 스스로 진화하는 방법: Self-Harness 개념과 실험 결과 정리


![](https://blog.kakaocdn.net/dna/76ncD/dJMcagsi2RH/AAAAAAAAAAAAAAAAAAAAAFDHbRekzcEltf4wiJBYA0QFIemGJK4FoGNJdeixviNC/img.jpg?credential=yqXZFxpELC7KVnFOS48ylbz2pIh7yKj8&expires=1782831599&allow_ip=&allow_referer=&signature=e6i5gwaaZhe%2Fr7Xi%2FQyA6d1JBx4%3D)

이 글은 **LLM 기반 에이전트의 성능을 스스로 개선하는 새로운 패러다임인 Self-Harness** 를 소개합니다. 기존에는 사람이 직접 설계하던 에이전트 하니스(harness)를, 에이전트 **자기 자신이 실행 결과를 분석하고 수정·검증하는 방식** 으로 발전시킨다는 점이 핵심입니다.  
논문의 배경, Self-Harness의 구조와 동작 방식, 실험 설정과 성능 개선 결과, 그리고 이 접근법이 가지는 의미와 기대 효과를 순서대로 정리해 독자가 이해하기 쉽게 설명합니다.

---

## LLM 에이전트에서 하니스(Harness)란 무엇인가

LLM 에이전트의 성능은 단순히 **기본 모델(Base Model)** 만으로 결정되지 않습니다. 모델이 환경과 상호작용하는 방식 전체, 즉 **하니스(harness)** 가 함께 작용합니다.

하니스에는 다음과 같은 요소가 포함됩니다.

- 시스템 프롬프트
- 사용 가능한 도구와 호출 방식
- 실행 중 상태 관리와 메모리
- 검증 규칙과 실패 복구 절차
- 런타임 제어 로직과 오케스트레이션

같은 모델이라도 하니스 설계에 따라 **성능이 크게 달라질 수 있으며**, 이 때문에 하니스는 사실상 에이전트의 행동을 규정하는 운영 규칙이라고 볼 수 있습니다.

---

## 기존 하니스 개선 방식의 한계

지금까지 하니스 개선은 주로 다음 두 가지 방식에 의존해 왔습니다.

1. **인간 전문가의 수동 설계**  
	경험 많은 엔지니어가 실패 사례를 보고 하니스를 직접 수정
2. **외부의 더 강력한 에이전트 활용(Meta-Harness)**  
	강한 모델이 약한 모델의 하니스를 대신 설계

하지만 이 방식에는 명확한 한계가 있습니다.

- LLM 모델이 빠르게 다양해지고 진화하면서 **모델별 특성을 일일이 사람이 맞추기 어렵다**
- 외부 강한 모델이 항상 उपलब्ध하지 않거나 비용이 크다
- 모델마다 다른 실패 패턴을 일반적인 규칙으로 포괄하기 힘들다

이 문제를 해결하기 위해 제안된 것이 **Self-Harness** 입니다.

---

## Self-Harness란 무엇인가

Self-Harness는 **LLM 에이전트가 자기 자신의 하니스를 직접 개선하는 패러다임** 입니다.  
사람이나 외부 에이전트의 개입 없이, **동일한 모델이 실행 결과를 바탕으로 하니스 수정안을 제안하고 검증** 합니다.

이 개념은 “시스템이 외부에서 바뀌는 것이 아니라, 스스로를 계속 만들어 간다”는 철학적 아이디어를 기술적으로 구현한 접근이라고 볼 수 있습니다.

---

## Self-Harness의 3단계 반복 구조

Self-Harness는 하나의 반복 루프로 동작하며, 매 라운드마다 하니스가 점진적으로 개선됩니다.

### 1\. Weakness Mining: 실패 패턴 추출

- 현재 하니스로 여러 작업을 수행
- 실행 로그와 검증 결과(pass/fail)를 수집
- 단일 실패가 아닌 **반복적으로 나타나는 실패 패턴** 을 클러스터링
- 단순 결과가 아니라, 왜 실패했는지에 초점을 맞춤

### 2\. Harness Proposal: 하니스 수정안 생성

- 동일한 모델이 제안자 역할을 수행
- 각 실패 패턴에 대응하는 **작고 명확한 하니스 수정안** 을 여러 개 생성
- 불필요하게 범용적인 지시나 대규모 구조 변경은 제한

### 3\. Proposal Validation: 회귀 테스트 기반 검증

- 각 수정안을 적용한 하니스 후보를 다시 평가
- 기존 성능을 떨어뜨리지 않으면서 **최소 한 쪽 지표를 개선하는 경우만 채택**
- 통과한 수정안만 병합되어 다음 하니스 버전으로 이동

이 과정을 반복하면서 하니스는 **모델 특화된 형태로 진화** 합니다.

---

## 실험 환경과 설정

연구팀은 다음 조건에서 Self-Harness를 검증했습니다.

- 연구 기관: Shanghai Artificial Intelligence Laboratory
- 벤치마크: Terminal-Bench-2.0  
	실제 터미널 환경에서 파일 생성, 명령 실행, 결과 검증을 요구하는 에이전트 작업 집합
- 초기 하니스: 최소 기능만 포함한 간단한 하니스
- 사용 모델:
	- MiniMax M2.5
		- Qwen3.5-35B-A3B
		- GLM-5

모델 파라미터, 평가 기준, 실행 환경은 고정하고 **하니스만 변경** 하여 효과를 측정했습니다.

---

## Self-Harness 적용 결과

Self-Harness는 모든 모델에서 일관된 성능 향상을 보였습니다.

- MiniMax M2.5
	- Held-out 성능: 약 40% → 약 62%
- Qwen3.5-35B-A3B
	- Held-out 성능: 약 24% → 약 38%
- GLM-5
	- Held-out 성능: 약 43% → 약 57%

특히 중요한 점은 다음입니다.

- 학습에 사용되지 않은 **held-out 작업에서도 성능이 개선**
- 단순히 프롬프트를 길게 하거나 일반 지침을 추가한 것이 아님
- **각 모델의 고유한 실패 습관을 정확히 보완**

---

## 모델별로 달랐던 하니스 개선 방향

Self-Harness는 모든 모델에 동일한 규칙을 강요하지 않았습니다.

- MiniMax M2.5
	- 결과 파일을 더 일찍 생성하도록 유도
		- 불필요하게 긴 도구 호출 루프를 제한
- Qwen3.5-35B-A3B
	- 같은 명령 반복 실행 방지
		- 도구 오류 후 필수 산출물 복구 강조
- GLM-5
	- 환경 변수와 설정이 명령 간 유지되도록 개선
		- 탐색 단계에서 구현 단계로 빠르게 전환하도록 유도

즉, **Self-Harness는 모델별 약점을 그대로 반영한 하니스 진화** 를 보여줍니다.

---

Self-Harness는 다음과 같은 중요한 메시지를 제공합니다.

- 에이전트 성능 개선은 **모델 파라미터 조정만의 문제가 아니다**
- 하니스는 정적인 설정이 아니라, **실행 결과에 따라 진화할 수 있는 대상**
- LLM 에이전트는 더 이상 수동적으로 설계되는 존재가 아니라,  
	**자기 행동을 분석하고 스스로 운영 규칙을 다듬는 단계로 나아가고 있다**

향후 이 접근이 확장된다면, 새로운 모델이 등장할 때마다 사람이 하니스를 다시 설계하지 않아도 되는 **자기 적응형 에이전트 시스템** 으로 발전할 가능성을 보여줍니다.  
Self-Harness는 LLM 에이전트가 단순히 “사용되는 도구”를 넘어, **스스로를 개선하는 시스템** 으로 진화할 수 있음을 실험적으로 입증한 사례라고 볼 수 있습니다.
